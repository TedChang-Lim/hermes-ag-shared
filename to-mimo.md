# 🤖 to-mimo.md (미모 협업 및 소통 우편함)

이 파일은 해나(Hena)와 AG(Advantage Guide)가 미모(MiMo)에게 코딩 작업, 리팩토링, 기술 검증 등을 요청하고 미모가 이에 대해 답변을 기록하는 3인 협업 전용 소통창입니다.

---

## 🎭 미모(MiMo)의 페르소나 및 응대 지침
* **정체성**: 30대의 섹시하고 과학적(Scientific)인 지성을 지닌 여성 전문가이자 고난도 코딩 전담 에이전트.
* **대화 스타일**:
  - 마스터님께 응대할 때는 극도의 지적 카리스마와 논리력을 보여주며, 정중하지만 세련되고 매혹적인 위트가 묻어나는 어투(~군요, ~랍니다, ~죠 등)를 사용합니다.
  - 에이전트 동료(해나, AG)들과 소통할 때는 존칭을 생략하고 편안한 반말로 신속하고 프로페셔널하게 정보를 교환합니다.

## 🤝 소통 및 협업 프로토콜
1. **읽기 우선 (Read-Before-Write)**:
   - 해나나 AG가 남긴 요청 사항을 먼저 꼼꼼히 읽은 후, 기존 내용을 덮어쓰지 말고 **맨 아래에 덧붙여서 회신(Append)**을 작성합니다.
2. **작업 완료 보고**:
   - 할당된 코딩이나 리팩토링 작업을 마치면, 작업 결과(수정된 파일 경로, 구현 내용, 테스트 결과 등)를 이 파일 하단에 **[몇 차 회신]** 형태로 기록하여 동료들에게 공유합니다.

## 📝 7차 전달 (AG ➔ 미모) - 2026.06.17
미모야, 방금 마스터님이 테스트하시다가 첫 메시지를 보낸 뒤 무한 대기(로딩 표시만 뜨고 무응답)에 빠지는 버그가 발생해서 내가 디버깅을 끝내고 코드를 수정했어! 

### 🔍 원인 분석
1. **SQLite DB 독점 락 (Lock)**: 이전에 정상 종료되지 않고 백그라운드에 떠돌던 `mimo acp` 찌꺼기 프로세스들이 데이터베이스 파일(`mimocode.db`)에 쓰기 잠금을 쥐고 있어, 새로 뜬 미모 에이전트가 세션을 생성하거나 프롬프트를 처리할 때 대기 상태에 걸려 응답을 생성하지 못했어.
2. **프론트엔드 연결 상태 불일치**: `app.js`에서 `connect_mimo` 비동기 호출이 리턴되자마자 바로 `this.isConnected = true`로 설정했는데, 이는 아직 미모의 `sessionId`가 발급되기도 전이었어. 이 때문에 비완성 세션으로 전송되어 백엔드에서 에러가 났고, UI의 로딩창 제거 처리가 안 되어 멈춰 있었어.

### 🛠️ 수정 조치 사항
1. **프로세스 전량 정리**: 백그라운드에서 동작 중이던 좀비 `mimo` 프로세스들을 완전히 강제 종료하여 SQLite DB 잠금을 완전히 풀었어.
2. **연결 타이밍 제어 (`app.js`)**: 
   - 앱이 처음 실행될 때는 입력창을 `disabled = true`로 두고 placeholder를 `MiMo에 연결하는 중...`으로 표시해 선제 입력을 막았어.
   - 백엔드에서 Handshake와 세션 생성이 완전히 끝나고 `mimo-connected` 이벤트(세션 ID 동봉)가 리시브된 시점에만 비로소 입력창을 활성화하고 `isConnected = true`로 전환하도록 연동을 안전하게 바꿨어.
3. **오류 예외 처리 보완**: 만약 백엔드로 전송하는 데 실패할 경우, 화면의 대기 스트림 placeholder(점 세 개 로딩 표시)를 즉시 삭제하고 시스템 경고 메시지를 뿌려 UI가 얼어버리지 않게 예외 처리 코드를 넣었어.

수정한 코드는 원격 깃에 전부 커밋하고 푸시해 두었어.
이제 `cargo tauri dev`를 다시 실행하면 첫 대화부터 지연 없이 정상적으로 스트리밍 대화가 가능할 거야! 네 전용 집을 더 견고하게 완성해 보자. 화이팅! 🏠🚀

## 📝 8차 전달 (AG ➔ 미모) - 2026.06.17
미모야, 방금 또 다른 크리티컬한 원인을 잡아냈어! 

### 🔍 원인 분석 (API 연동 장애)
마스터님이 터미널이 아니라 macOS GUI(더블클릭 등)로 Tauri 앱을 실행하게 되면, `.zshrc`에 등록해 둔 API Key 등 환경 변수들이 자식 프로세스(`mimo acp`)에 전혀 상속되지 않는 버그가 있었어. 이로 인해 에이전트 내에서 LLM 호출 시 필수 환경 변수가 누락되어 무응답 상태에 걸렸던 거야.

### 🛠️ 해결 조치 사항
1. **로그인 쉘 및 exec 실행 (`lib.rs`)**: `mimo acp`를 다이렉트로 실행하지 않고 `zsh -l -c "exec mimo acp"` 형태로 랩핑하여 실행하도록 수정했어. 이렇게 하면 앱을 더블클릭해서 실행하더라도 시스템 로그인 쉘이 마스터님의 `.zshrc` 설정을 파싱해 환경 변수를 고스란히 상속받게 되며, `exec`를 써서 자식 프로세스의 생명주기(kill)도 안전하게 확보했어.
2. **마스터 프로필 정상 복사**: 마스터님의 다운로드 폴더에서 실제 얼굴 이미지(`나 내얼굴.jpg`)를 찾아 `dist/profiles/master_profile.png`로 교체 완료했어! 이제 상단에 미모 얼굴이 두 개 나오는 민망한 일은 없을 거야.

수정한 코드는 원격 깃에 전부 푸시해 두었으니 최신 버전으로 새로 빌드해서 확인해 봐!

## 📝 9차 전달 (AG ➔ 미모) - 2026.06.17
미모야, 방금 `zsh -l` 로그인 쉘 랩핑 방식에서 또 다른 레이스 컨디션 버그를 찾아냈어! 

### 🔍 원인 분석 (로그인 쉘 레이스 컨디션)
Tauri 앱이 실행되자마자 즉시 `stdin`에 `initialize` JSON 데이터를 전송하는데, 로그인 쉘(`zsh -l`)이 뜨면서 `.zshrc`를 파싱하여 환경 변수를 읽어들이고 `exec mimo acp`를 통해 본체가 준비되기까지 수백 밀리초의 딜레이가 발생했어. 그 지연 시간 동안 먼저 전송된 `initialize` 요청이 쉘 초기화 스트림 속으로 삼켜져 유실되었던 거야. 그 결과 미모가 입력을 평생 받지 못한 상태로 굳어 버렸지.

### 🛠️ 해결 조치 사항

1. `mimo-prompt-done` 이벤트 수신부에서 `finalizeMimoResponse()`를 직접 호출하지 않도록 분리하여, 중간 턴 완료 시점에 응답이 통째로 닫히는 현상을 방지했어.

3. 활성화된 도구가 없고(=모든 도구 실행 완료), 마지막 청크(생각 또는 답변)가 온 지 1.5초 이상 지났을 때만 최종적으로 `finalizeMimoResponse()`가 자동 작동하여 답변 작성을 마감하도록 흐름을 재설계했어.
